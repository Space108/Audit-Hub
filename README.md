# Audit-Hub

**Audit-Hub** — аналитическое приложение для **предварительной приёмки текста ТЗ перед разработкой**: по чек-листу показывает пробелы в формулировках, отдельно проверяет **в том же документе**, связаны ли описанные сущности с реализацией и источниками данных, и при прохождении критичных правил готовит **тест-кейсы**. Работает с текстом ТЗ (ввод вручную или файл `.txt` / `.docx` / `.pdf` с текстовым слоем); анализ через LLM, без ключа — упрощённый демо-режим.

## Возможности

- Загрузка чек-листа из `json`, `csv`, `xlsx/xls`, `txt`, `md`.
- Добавление новых правил через UI без перезапуска приложения.
- Ввод текста ТЗ вручную или загрузка файлов `.txt` / `.docx` / `.pdf`.
- Автоподстановка тестового ТЗ из `sample_tz.txt`.
- Проверка каждого правила чек-листа через LLM API или встроенный демо-режим.
- Три статуса проверки: `Найдено`, `Частично`, `Не найдено`.
- Доказательная база по каждому правилу: цитаты, что не раскрыто и рекомендация.
- **Сквозной анализ связности** (только live LLM): сущности в тексте ТЗ и их связь с описанием реализации в этом же тексте; статусы «Связано» / «Частично» / «Нет реализации».
- Отдельные метрики: `Strict score`, `Coverage score`, `Blocker score`.
- Компактная таблица с найденными, частично раскрытыми и отсутствующими пунктами чек-листа.
- Генерация тест-кейсов, если все `blocker`-правила пройдены.
- Выгрузка актуального чек-листа и тест-кейсов в JSON.
- Выгрузка Excel-отчета: сводка, анализ по правилам, пробелы, **traceability**, тест-кейсы.

## Установка

Рекомендуется использовать виртуальное окружение.

```powershell
python -m venv .venv
.\.venv\Scripts\Activate.ps1
pip install streamlit pandas openai python-docx python-dotenv pypdf openpyxl
```

## Запуск

```powershell
streamlit run app.py
```

После запуска Streamlit откроет приложение в браузере.

## Режимы работы

### Демо-режим

Если API-ключ LLM-провайдера не задан, приложение автоматически использует демо-режим. В этом режиме проверка работает по простым эвристикам и подходит только для демонстрации интерфейса и базового сценария.

Демо-режим можно включить или выключить в sidebar.

### LLM-режим

Для полноценного анализа задайте ключ в переменной окружения:

```powershell
$env:LLM_API_KEY_ENV="GROQ_API_KEY_BACKUP_2"
$env:GROQ_API_KEY_BACKUP_2="your_llm_api_key"
streamlit run app.py
```

Или сохраните ключ локально в `.env` (файл не попадет в git):

```env
LLM_PROVIDER=Groq
LLM_API_KEY_ENV=GROQ_API_KEY_BACKUP_2
LLM_BASE_URL=https://api.groq.com/openai/v1
LLM_MODEL=your_model
GROQ_API_KEY_BACKUP_2=your_groq_api_key
```

Модель можно переопределить через переменную окружения:

```powershell
$env:LLM_MODEL="your_model"
streamlit run app.py
```

### Какие LLM совместимы

Приложение общается с моделью **только через OpenAI-compatible API**: тот же контракт, что использует официальный Python-клиент `openai` (метод chat completions, путь вида `.../v1/chat/completions` на стороне провайдера). В запросе включается режим **ответа в JSON** (`response_format: json_object`), поэтому endpoint и сама модель должны это поддерживать (или прокси должен корректно это эмулировать).

**Подойдёт**, если вы можете указать в настройках:

- `LLM_BASE_URL` — базовый URL совместимого API (как в документации провайдера для OpenAI SDK);
- ключ в переменной, имя которой задано в `LLM_API_KEY_ENV`;
- `LLM_MODEL` — идентификатор модели у этого провайдера.

**Типичные примеры** (не исчерпывающий список: важен именно совместимый API, а не бренд):

| Провайдер / сценарий | Заметка |
|----------------------|---------|
| **Groq** | Дефолт в примере `.env`; быстрые открытые модели. |
| **OpenAI** | `https://api.openai.com/v1` и модели вроде `gpt-4o`, `gpt-4o-mini`. |
| **Azure OpenAI** | Нужен **base URL и модель** из портала Azure для OpenAI-compatible вызовов (как в их инструкции для Chat Completions). |
| **OpenRouter, Together AI, Fireworks** и др. | Обычно дают свой `base_url` и список моделей под тот же клиент. |
| **Локально или свой сервер** | Например прокси **LiteLLM**, **vLLM** с OpenAI-совместимым сервером, **Ollama** с включённой совместимостью — если в ответе соблюдается ожидаемая структура и доступен JSON-режим. |

Если у сервиса **другой протокол** (не тот же chat completions / не тот же формат ключа и URL), приложение из коробки к нему не подключится — нужен отдельный шлюз с OpenAI-compatible входом или доработка кода.

## Формат checklist.json

Файл должен содержать объект с массивом `rules`.

```json
{
  "rules": [
    {
      "id": "1.1",
      "category": "Глоссарий",
      "name": "Расшифровка терминов",
      "description": "В тексте должны быть явно расшифрованы специфичные аббревиатуры.",
      "criticality": "warning"
    }
  ]
}
```

Поддерживаемые уровни критичности:

- `blocker` - критичное правило, без прохождения которого нельзя генерировать тест-кейсы.
- `warning` - важное замечание, не блокирует генерацию тест-кейсов.
- `info` - информационное правило.

## Форматы загрузки чек-листа

Поддерживаются:

- `JSON` (рекомендуется)
- `CSV`
- `XLSX/XLS`
- `TXT/MD` (строка = правило)

Для `CSV/XLSX` ожидаются колонки (можно на русском или английском):

- `id` / `rule_id`
- `category` / `категория`
- `name` / `правило` / `название`
- `description` / `описание`
- `criticality` / `критичность` / `priority`

Если явные названия колонок не найдены, приложение пробует использовать первые колонки файла как `category`, `name`, `description`, `criticality`.

Для `TXT/MD` используйте формат строки:

```text
category | name | description | criticality
```

## Основной сценарий

1. Запустите приложение.
2. При необходимости загрузите `checklist.json` в sidebar.
3. Добавьте новые правила через форму в sidebar.
4. Вставьте текст ТЗ или загрузите `.txt` / `.docx` / `.pdf`.
5. Нажмите `Проверить ТЗ`.
6. Изучите таблицу результатов, цитаты, рекомендации и метрики полноты.
7. Если все `blocker`-правила найдены, нажмите `Сгенерировать тесты`.
8. Запустите **Сквозной анализ** (в live-режиме LLM), чтобы получить матрицу связности сущностей и реализации.
9. Скачайте тест-кейсы в JSON и/или полный Excel-отчет (включая лист `Traceability`).
10. В sidebar раздел **Очистка**: «Только результаты» — сброс проверок без удаления текста ТЗ; «Всё: ТЗ и результаты» — полный сброс сессии.

## Результат проверки

После проверки приложение показывает:

- что покрыто в ТЗ;
- какие правила раскрыты частично или не найдены;
- комментарий ИИ с цитатой, причиной и рекомендацией;
- `Strict score` - доля полностью найденных правил;
- `Coverage score` - доля полностью или частично покрытых правил;
- `Blocker score` - доля полностью найденных blocker-правил;
- после запуска сквозного анализа (live LLM) — матрицу связности сущностей с описанием реализации **в том же тексте ТЗ**;
- таблицу тест-кейсов после успешного прохождения blocker-правил.

## Сквозной анализ

Анализ выполняется **по одному загруженному тексту ТЗ** (не по связке документов в репозитории). Модель универсально ищет в тексте сущности и проверяет, есть ли **в этом же тексте** достаточное описание реализации или источников. Класс сущностей, которые могут быть выделены, включает, например:

- поля и колонки;
- показатели;
- формулы и алгоритмы;
- фильтры и условия;
- роли и права доступа;
- интеграции и внешние системы;
- источники данных, документы, справочники, регистры, таблицы, API;
- пользовательские действия, статусы и бизнес-процессы.

Для каждой сущности определяется статус:

- `Связано` - есть требование и описание реализации;
- `Частично` - реализация описана неполно;
- `Нет реализации` - сущность заявлена, но источник, алгоритм или правило реализации не описаны.

Результат сквозного анализа добавляется в Excel-отчет на лист `Traceability`.

## Примечания

- Изменения чек-листа через UI хранятся в памяти текущей сессии Streamlit.
- Чтобы сохранить измененный чек-лист, используйте кнопку скачивания в sidebar.
- DOCX-файлы читаются по тексту параграфов, без анализа таблиц и вложенных объектов.
- PDF-файлы читаются только по текстовому слою. Сканированные PDF не анализируются, потому что OCR может исказить ТЗ и сделать проверку недостоверной.
